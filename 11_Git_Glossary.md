## Core Git Concepts

### Blob
A Git object type that stores the raw content of a single file. The name stands for Binary Large Object. Identified by its SHA hash. Does not store filename or permissions — those live in the tree object.

### Branch
A lightweight, movable pointer to a specific commit. Creating a branch is nearly free — Git just creates a small file containing a commit hash. Branches diverge when commits are added to one but not the other.

### Commit
A snapshot of the entire tracked file tree at a specific moment. Contains: a reference to a tree object, references to parent commit(s), author/committer info, timestamp, and message. Immutable — once created, its content never changes (though its reachability can).

### Commit Graph
The Directed Acyclic Graph (DAG) formed by all commits in a repository. Each commit points to its parent(s). Forms the complete version history. Branches and tags are simply named pointers into this graph.

### Commit Hash (SHA)
A 40-character hexadecimal string (e.g., `a3f5c1d2e8b9f4a1c7d3e6b2f9a4c1d8e3f7b2a1`) uniquely identifying a commit. Generated by hashing the commit's content, parent hash, timestamp, and author. Can be shortened to the first 7+ characters when unambiguous.

### HEAD
A special pointer indicating the current position in the repository — usually pointing to the currently checked-out branch, which in turn points to the latest commit on that branch. In detached HEAD state, HEAD points directly to a commit hash rather than a branch.

### Index
See **Staging Area**.

### Object
The fundamental storage unit in Git. Four types exist: blob (file content), tree (directory listing), commit (snapshot metadata), and tag (named reference with metadata). All objects are content-addressed: their SHA hash is derived from their content.

### Object Store
The `.git/objects/` directory where all Git objects (blobs, trees, commits, tags) are stored as compressed files. Forms the complete historical record of the repository.

### Repository (Repo)
A directory tracked by Git, containing the project files (working directory) plus the `.git/` folder that stores the full history. A "bare repository" contains only the `.git/` contents with no working directory, used for hosting (e.g., on GitHub's servers).

### Snapshot
Git's mental model for a commit — a complete picture of all tracked files at a moment in time, not a list of changes (diffs). Git deduplicates unchanged files via shared blob references for storage efficiency.

### Staging Area (Index)
A preparatory zone between the working directory and the repository. Files are "staged" with `git add` before being committed. Allows crafting precise commits by choosing exactly which changes to include. Stored in `.git/index`.

### Tree
A Git object representing a directory — contains a list of filenames with their permissions, each mapped to either a blob (file) or another tree (subdirectory). The root tree of a commit represents the project's entire file structure at that point.

### Working Directory (Working Tree)
The actual files on your disk that you edit. One of Git's three zones (working directory → staging area → repository). Changes here are "unstaged" until `git add` is run.

## Git Commands & Operations

### `git add`
Move changes from the working directory to the staging area. Does not create a commit.

### `git bisect`
Binary search through commit history to find which commit introduced a bug. Requires marking commits as "good" or "bad" until the culprit is isolated.

### `git blame`
Annotate each line of a file with the commit hash and author that last modified it. Used to trace when and why a line was changed.

### `git cherry-pick`
Apply the changes from a specific commit onto the current branch, creating a new commit with the same changes but a new hash. Does not move or delete the original commit.

### `git clean`
Remove untracked files from the working directory. Destructive — cannot be undone. Always run `git clean -n` (dry run) first.

### `git clone`
Create a local copy of a remote repository, including all history. Automatically sets `origin` to the source URL and checks out the default branch.

### `git commit`
Create a permanent snapshot from the contents of the staging area. Records who made the change, when, and why (the commit message).

### `git diff`
Show changes between zones (working directory, staging, commits). `git diff` = unstaged changes. `git diff --staged` = staged changes. `git diff branch1..branch2` = between two branches.

### `git fetch`
Download commits and objects from a remote repository without modifying local branches or the working directory. Updates remote-tracking branches (e.g., `origin/main`).

### `git filter-repo`
Modern tool for rewriting repository history (replacing deprecated `git filter-branch`). Used to remove files from all history, rename paths, or change author information across commits.

### `git fsck`
Verify the integrity of the Git object database. Can locate "dangling" (unreachable) commits — useful for recovering lost work after destructive operations.

### `git gc` (Garbage Collection)
Clean up unnecessary files and optimize the repository. Packs loose objects into packfiles, removes expired reflog entries, prunes unreachable objects.

### `git init`
Initialize a new Git repository in the current directory by creating the `.git/` folder.

### `git log`
Display the commit history. Many flags control format: `--oneline` (compact), `--graph` (visual branches), `--all` (include all branches), `-p` (show diffs).

### `git merge`
Integrate changes from one branch into another. May create a merge commit (three-way merge) or simply advance a pointer (fast-forward). Can result in conflicts.

### `git pull`
Shorthand for `git fetch` followed by `git merge` (or `git rebase` if configured). Brings the current branch up to date with its remote counterpart.

### `git push`
Send local commits to a remote repository. Rejected if the remote has commits the local doesn't (you must pull first, unless force-pushing).

### `git rebase`
Move or replay commits onto a different base commit. Rewrites commit history (new hashes). Used to linearize history or update a feature branch with latest changes from main.

### `git reflog`
Record of all movements of HEAD and branch pointers. The primary recovery tool for "lost" commits from resets, deleted branches, or bad rebases. Entries expire after ~90 days.

### `git reset`
Move the current branch pointer to a different commit. Three modes: `--soft` (keep staged), `--mixed` (keep in working dir, default), `--hard` (discard all changes). Destructive with `--hard` when changes are not committed.

### `git restore`
Discard changes in the working directory (`git restore file`) or unstage changes (`git restore --staged file`). Introduced in Git 2.23 to split the responsibilities of `git checkout`.

### `git revert`
Create a new commit that undoes the changes of a specified commit. Safe for shared/published history because it adds to history rather than rewriting it.

### `git rm`
Remove a file from the working directory and stage that removal. `git rm --cached` removes only from tracking (keeps file on disk).

### `git show`
Display information about a specific Git object (commit, tag, tree, blob). For commits, shows metadata plus the diff.

### `git stash`
Temporarily shelve uncommitted changes (staged and unstaged) to a stack, leaving a clean working directory. Restored later with `git stash pop` or `git stash apply`.

### `git status`
Show the state of the working directory and staging area: which files are tracked/untracked, modified, or staged.

### `git switch`
Check out a branch (or create one with `-c`). Introduced in Git 2.23 to separate the branch-switching responsibility from the overloaded `git checkout` command.

### `git tag`
Create a named pointer to a specific commit. **Lightweight tags** are just pointers. **Annotated tags** include metadata (tagger, date, message) and are stored as full objects. Tags do not move when new commits are added (unlike branches).

## Branching & Merging Terms

### Base Branch
The branch that another branch was created from, or the target branch of a PR. In a PR, the "base" is what you're merging *into*.

### Compare Branch
The source branch of a PR — the branch containing changes you want to merge into the base.

### Conflict (Merge Conflict)
Occurs when two branches modify the same lines of the same file differently, and Git cannot automatically determine which version to use. Must be resolved manually before the merge or rebase can complete.

### Detached HEAD
State where HEAD points directly to a commit rather than to a branch name. Commits made in this state have no branch label and risk being lost when switching away. Fix with `git switch -c new-branch-name`.

### Diverge
When two branches have each received commits that the other doesn't have. A diverged branch cannot be fast-forward merged — requires a three-way merge or rebase.

### Fast-Forward Merge
A merge where the target branch pointer simply moves forward to the source branch tip — possible only when the target hasn't received any commits since the source branch was created (no divergence). Creates no merge commit.

### Feature Branch
A short-lived branch created specifically for developing a single feature, bug fix, or unit of work. The fundamental unit of Git collaboration. Merged and deleted after the work is complete.

### Hotfix Branch
A branch created directly from the production branch (`main` or `production`) to fix a critical bug in production. Merged back to both production and the development branch.

### Long-lived Branch
A branch that persists indefinitely, like `main` or `develop`. In Git Flow, `develop` is long-lived. In GitHub Flow, only `main` is long-lived.

### Merge Commit
A commit with two parents, created when two diverged branches are merged. Documents the merge event in history. Has its own hash and timestamp.

### Rebase
See **git rebase** above. When used as a noun: the operation of replaying commits onto a different base. "After the rebase, the branch had a linear history."

### Short-lived Branch
See **Feature Branch**. Branches that are created, used, and deleted — the norm in GitHub Flow and trunk-based development.

### Squash Merge
Combining all commits from a source branch into a single commit before merging into the target branch. Produces a clean history at the expense of losing individual commit context from the feature branch.

### Three-Way Merge
A merge algorithm that uses three snapshots: the common ancestor, the tip of branch A, and the tip of branch B. Used when branches have diverged. May produce a merge commit.

### Tracking Branch
A local branch configured to correspond to a remote branch. Enables bare `git push` / `git pull` without specifying remote and branch names. Set with `git push -u` or `git branch --set-upstream-to`.

## Remote & Collaboration Terms

### Clone
A local copy of a remote repository containing the full history.

### Fetch
Download remote changes to update remote-tracking branches, without affecting local branches.

### Force Push (`--force`)
Overwrite a remote branch with your local state, ignoring the remote's current commits. Dangerous on shared branches. Prefer `--force-with-lease`.

### `--force-with-lease`
A safer force push that fails if the remote branch has received commits since your last fetch. Prevents accidentally overwriting a teammate's work.

### Origin
The conventional default name for the remote repository a project was cloned from. Set automatically by `git clone`.

### Pull
`git fetch` + `git merge` (or `git rebase`). Brings local branch up to date with its remote counterpart.

### Push
Send local commits to a remote repository.

### Remote
A named reference to another Git repository (usually on a server like GitHub). Stored as a URL alias in `.git/config`. Multiple remotes can be configured (e.g., `origin`, `upstream`, `heroku`).

### Remote-Tracking Branch
A local, read-only reference to the state of a branch in a remote repository as of the last fetch. Named `remote/branch` (e.g., `origin/main`). Not a branch you commit to — only updated by fetch/pull.

### Upstream (remote)
Conventionally, the original repository that a fork was created from. You pull updates from `upstream` and push work to `origin` (your fork).

### Upstream (branch)
The remote branch that a local branch is configured to track. "The upstream of `feature-login` is `origin/feature-login`."

## GitHub-Specific Terms

### CODEOWNERS
A file (`.github/CODEOWNERS`) that defines which GitHub users or teams are automatically requested as reviewers for changes to specific files or directories.

### Contribution Graph
The grid of green squares on a GitHub profile showing daily contribution activity over the past year. Counts commits to default branches, PR openings, issue creation, and review activity.

### Dependabot
GitHub's automated dependency update service. Opens PRs to update outdated or vulnerable dependencies based on configured schedules.

### Draft PR
A pull request marked as not ready for review. CI runs, but reviewers are not officially requested. Used for work-in-progress or early feedback.

### Fork
A copy of a repository under a different GitHub account. Used to contribute to projects without direct write access. Maintains a relationship with the original ("upstream") repo.

### Gist
GitHub's service for sharing small code snippets or notes. Backed by a full Git repository.

### GitHub Actions
GitHub's built-in CI/CD and automation platform. Workflows are defined in YAML files in `.github/workflows/`. Triggered by Git events (push, PR, tag, schedule).

### GitHub CLI (`gh`)
The official command-line interface for GitHub. Enables creating repos, PRs, issues, and releases from the terminal without opening a browser.

### GitHub Discussions
A forum-style feature within a repository for Q&A, announcements, and community conversations — separate from the issue tracker.

### GitHub Pages
Free static website hosting from a GitHub repository. Serves files from a specified branch or folder.

### GitHub Releases
A GitHub feature for packaging and distributing software versions. Built on top of Git tags. Supports release notes and binary file attachments.

### Issue
A trackable unit of work in a GitHub repository. Used for bug reports, feature requests, tasks. Can be assigned, labeled, linked to milestones and projects, and referenced by commits and PRs.

### Label
A tag on an issue or PR used for categorization (e.g., `bug`, `enhancement`, `good first issue`).

### Milestone
A grouping of issues and PRs toward a shared goal, with an optional due date. Used for sprint or version planning.

### Personal Access Token (PAT)
A credential used in place of a password for GitHub API access or HTTPS Git operations. Generated in GitHub Settings → Developer settings.

### Pull Request (PR)
A GitHub feature for proposing code changes from one branch to another. Provides a space for code review, discussion, CI status, and controlled merging. Not a Git feature — it's GitHub's layer on top of Git.

### Required Status Check
A CI check that must pass before a PR can be merged into a protected branch. Configured in branch protection rules.

### Review
A formal assessment of a PR's code changes. Three outcomes: Comment (no verdict), Approve, or Request Changes.

### Secret (GitHub Secret)
An encrypted value stored in a repository or organization, accessible to GitHub Actions workflows but not visible in logs or to unauthorized users.

### Star
GitHub's bookmarking/favoriting mechanism for repositories. Also functions as a social signal of project popularity.

### Webhook
An HTTP callback triggered by GitHub events (push, PR, issue). Used to notify external services (Slack, deployment systems) of repository activity.

## CI/CD & Deployment Terms

### Artifact (CI)
A file produced by a CI build (compiled binary, test report, Docker image, coverage report) that is stored and potentially deployed or attached to a release.

### Blue/Green Deployment
A release strategy where two identical production environments exist (blue = current, green = new). Traffic is switched from blue to green after the new version is validated. Enables instant rollback.

### Canary Deployment
Gradually routing a small percentage of production traffic to a new version before full rollout. Limits blast radius if the new version has issues.

### CD (Continuous Deployment / Continuous Delivery)
- **Continuous Delivery**: every passing build is *ready* to deploy to production (requires manual trigger)
- **Continuous Deployment**: every passing build is *automatically* deployed to production

### CI (Continuous Integration)
The practice of frequently merging code changes to a shared branch, with automated builds and tests run on each merge. Detects integration problems early.

### Environment (deployment)
An isolated runtime context (development, staging, production) with its own configuration, database, and deployment. GitHub Environments add protection rules and scoped secrets.

### Pipeline
The automated sequence of steps executed when code is pushed: install → lint → test → build → deploy. Defined in GitHub Actions workflow YAML.

### Preview Deployment
An ephemeral deployment created automatically for each pull request. Allows reviewers to test changes in a real environment before merging.

### Runner
The machine (virtual or physical) that executes a GitHub Actions job. GitHub provides hosted runners (Ubuntu, Windows, macOS). Self-hosted runners run on your own infrastructure.

### Smoke Test
A quick, high-level test run against a live deployment to verify the application is fundamentally functional (e.g., "does the homepage load? does login work?"). Run after deployment before marking it successful.

### Workflow (GitHub Actions)
A YAML-defined automation triggered by GitHub events. Composed of one or more jobs. Stored in `.github/workflows/`.

## Workflow & Process Terms

### Blameless Post-Mortem
A structured review of an incident or outage focused on systemic causes rather than individual blame. Documents timeline, root cause, impact, and action items.

### Code Review
The practice of having one or more developers read and evaluate another developer's code changes before they are merged. The primary quality gate in professional teams.

### Conventional Commits
A commit message specification defining a structured format: `type(scope): description`. Enables automated changelogs and semantic versioning.

### Feature Flag (Feature Toggle)
A configuration switch that enables/disables a feature at runtime without deploying new code. Enables trunk-based development by allowing incomplete features to be committed but hidden.

### Git Flow
A branching strategy by Vincent Driessen using `main`, `develop`, `feature/*`, `release/*`, and `hotfix/*` branches. Suited to scheduled releases with explicit versioning.

### GitHub Flow
A simplified branching strategy: branch from `main`, open PR, merge to `main`, deploy. Suited to continuous delivery and web applications.

### Golden Rule of Rebasing
Never rebase commits that have been pushed to a shared remote branch and may have been fetched by others. Doing so rewrites history and forces others to reconcile diverged histories.

### Semantic Versioning (SemVer)
A versioning scheme: `MAJOR.MINOR.PATCH`. MAJOR = breaking changes, MINOR = new backward-compatible features, PATCH = backward-compatible bug fixes.

### Trunk-Based Development
A strategy where all developers commit to a single main branch ("trunk") at least daily. Long-lived branches are eliminated. Requires feature flags and strong CI.

## File & Repository Structure Terms

### `.git/`
The hidden directory at the root of a Git repository containing all version control data: object store, refs, config, hooks, reflog, index. This directory *is* the repository.

### `.gitattributes`
A file that defines per-path attributes affecting Git behavior: line ending handling (CRLF/LF), merge strategies, diff handling, and binary vs text classification.

### `.github/`
A conventional directory for GitHub-specific configuration files: Actions workflows, issue templates, PR templates, CODEOWNERS, Dependabot config.

### `.gitignore`
A file specifying patterns of files and directories Git should not track. Applied to untracked files — does not un-track already-committed files.

### `CHANGELOG.md`
A document recording notable changes in each version of a project. Often auto-generated from conventional commit messages using tools like `semantic-release`.

### `CONTRIBUTING.md`
A file describing how to contribute to a project: code style, branch naming, PR process, development setup. Standard in open source projects.

### `LICENSE`
A file specifying the legal terms under which the software can be used, modified, and distributed.

### `README.md`
The primary documentation file for a project. Rendered on the GitHub repository homepage. Should include: what the project is, how to install it, how to use it, and how to contribute.

## Advanced & Internal Terms

### Blob (Git object)
See **Blob** in Core Git Concepts.

### DAG (Directed Acyclic Graph)
The data structure underlying Git's commit history. "Directed" because edges (parent pointers) have direction. "Acyclic" because there are no cycles — you can't follow parent pointers and return to the same commit.

### Dangling Commit
A commit object that is not reachable from any branch, tag, or ref. Created by operations like `git reset --hard` or deleting a branch. Visible via `git fsck --lost-found`. Eligible for garbage collection.

### Garbage Collection (`git gc`)
The process of cleaning up unreachable objects and packing loose objects into packfiles. Runs automatically or can be triggered manually. Dangling commits become permanently deleted after GC.

### GPG Signing
Cryptographic signing of commits using a GPG key pair. The "Verified" badge on GitHub commits indicates a valid signature matching the committer's registered public key.

### Loose Object
A Git object stored as an individual compressed file in `.git/objects/`. Converted to packfile format by `git gc` for efficiency.

### Packfile
A single file in `.git/objects/pack/` containing many Git objects, compressed together using delta encoding. Far more space-efficient than loose objects for large repositories.

### Reflog
See **git reflog** in Git Commands. The safety net record of all HEAD and branch movements, enabling recovery of "lost" commits.

### SHA-1 / SHA-256
The hash algorithms used to generate commit identifiers. Git historically used SHA-1 (40 hex chars). SHA-256 support (64 hex chars) is available in newer Git versions for improved collision resistance.

### Submodule
A Git repository embedded inside another Git repository at a specific commit reference. Used to include external dependencies or shared libraries while maintaining separate version histories.

### Worktree
An additional working directory checked out from the same repository. Allows multiple branches to be checked out simultaneously in different directories without switching or stashing.
